# AI System Analyst
## AIと一緒にプロジェクトを育てるワークスペース

Version 0.1

---

# 1. ビジョン

本プロジェクトは、システム開発における要件定義・設計工程をAIとの対話によって支援するWebアプリケーションを構築することを目的とする。

利用者は設計書を書く必要はなく、AIとの自然な対話を行うだけで、要件定義・基本設計・詳細設計が段階的に構築される。

本システムは「AIチャット」ではなく、「AIシステムアナリスト」として利用者の思考整理と設計支援を行う。

---

# 2. コンセプト

> AIと一緒にプロジェクトを育てるワークスペース

プロジェクトは完成品を一度に作るものではない。

利用者とAIが対話を重ねることで、

- 要件が整理される
- 設計が育つ
- ドキュメントが育つ
- システムが育つ

その過程そのものを支援する。

---

# 3. 開発思想

本システムでは以下の思想を採用する。

## AIは文章を書かせるものではない

AIは思考を整理する存在である。

利用者へ質問し、

回答を整理し、

構造化し、

設計書へ変換する。

---

## ドキュメントは成果物ではない

ドキュメントはプロジェクトの知識ベースである。

会話を続けるほど、

設計書は成長していく。

---

## AIはシステムアナリストである

AIは単なる質問応答ではない。

不足している情報を検知し、

次に確認すべき内容を提案し、

利用者の考えを整理する。

# 14.補足
ここまで一緒に考えてきて、このプロジェクトの本質は**「AIとの対話で設計書を作る」**ことではないと感じています。

本質は、

「人間の頭の中にある曖昧なアイデアを、AIが対話を通じて構造化し、プロジェクトの知識資産へ変換すること」

です。

つまり、最終成果物はPDFやMarkdownではなく、**プロジェクトの知識（Knowledge Base）**です。

Markdown、PDF、基本設計書、詳細設計書は、その知識から生成されるビュー（見せ方）の一つに過ぎません。

この考え方を採用すると、アプリ全体の設計思想が一貫します。

ユーザーは対話するだけ。
AIは知識を整理し、足りない点を質問する。
システムは知識を蓄積する。
各種ドキュメントは知識から自動生成する。

この思想をプロジェクトの憲章として最初に定義しておけば、今後機能を追加しても方向性がぶれにくい、強いプロダクトになると思います。
